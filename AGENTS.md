# TravelMind Agent Guidelines

本文件是 TravelMind 项目的智能体协作规则，适用于整个仓库。后续任何智能体或开发者修改代码时，都应优先遵守本文件。

## 项目定位

TravelMind 是一个 Python 旅行智能体项目，当前核心形态是 ReAct 风格主循环：

- `main.py` 负责用户请求、prompt history、模型调用、Action 解析和工具执行。
- `client/` 放模型客户端封装。
- `tools/` 放可被智能体调用的旅行工具。
- `prompts/` 放系统提示词和记忆提示词。
- `memory/` 放长期记忆模块。
- `test/` 放测试代码。
- `docs/superpowers/` 放设计文档和实施计划。

## 通用开发规则

- 优先保持改动小而清晰，只修改完成当前任务必需的文件。
- 不要重构无关代码，不要顺手格式化整仓库。
- 不要回滚、覆盖或删除用户已有改动，除非用户明确要求。
- 新增功能必须有对应测试；修复 bug 必须先补充能暴露问题的测试。
- 优先使用 Python 标准库；确实需要第三方依赖时，先说明理由和影响。
- 代码应兼容当前项目环境。当前环境可能是 Python 3.9，因此使用 `str | None` 等新语法时，需要配合 `from __future__ import annotations`。
- 不要提交 `.env`、缓存目录、虚拟环境、`__pycache__/`、`.pytest_cache/` 等本地生成内容。
- 不要在代码或文档中写入真实 API Key、Token、密码或其他敏感信息。

## 函数与类注释规则

新写或修改的非测试 Python 类、公共函数、公共方法必须添加简洁中文 docstring。

docstring 应说明：

- 这个类或函数负责什么。
- 重要参数的含义。
- 返回值的含义。
- 是否会读写文件、调用网络、修改状态或产生其他副作用。

示例：

```python
def maybe_update_memory(self, user_input: str) -> bool:
    """根据用户输入判断并更新长期记忆。

    Args:
        user_input: 当前轮用户的原始输入。

    Returns:
        如果写入或更新了记忆文件，返回 True；否则返回 False。

    Side Effects:
        可能创建或修改 `.memory/` 下的记忆文件和 `index.md`。
    """
```

私有辅助函数也建议添加 docstring，尤其当它们包含文件解析、索引修复、分类判断或不明显的业务规则时。非常短且语义完全显然的私有函数可以不写，但不能牺牲可读性。

测试函数通常不需要 docstring，测试名称应直接表达行为。

## 测试规则

- 使用 `pytest`。
- 测试命名应描述行为，例如 `test_budget_input_writes_budget_and_index`。
- 涉及文件系统的测试必须使用 `tmp_path`，不得读写真实项目 `.memory/`。
- 记忆模块测试至少覆盖：
  - 是否值得记忆的判断。
  - 分类写入。
  - `.memory/index.md` 创建和更新。
  - 索引修复。
  - 相关记忆加载。
- 每次改动后至少运行相关测试；完成任务前运行目标测试集合。

## 记忆模块规则

`memory/` 是长期记忆功能的唯一实现位置。第一版记忆功能保持程序层自动记忆，不在主循环中直接让模型读写文件。

- 记忆文件必须位于项目根目录 `.memory/`。
- `.memory/index.md` 是记忆系统唯一入口。
- 读取、写入、修复记忆前必须通过索引语义。
- 新增、修改、删除、合并或重命名记忆文件后，必须同步更新 `index.md`。
- 不值得长期记忆的信息不得写入 `.memory/`。
- 敏感信息默认不记忆，除非用户明确要求并且没有安全风险。
- 不要绕过 `MemoryManager` 直接操作 `.memory/` 文件，除非是在测试中构造场景。

## Prompt 规则

- `prompts/AGENTS_SYSTEM_PROMPT.py` 负责智能体主系统提示词。
- `prompts/MEMORY_PROMPT.py` 负责记忆功能提示词说明。
- 修改 prompt 时，应保持输出格式约束清晰，尤其是当前 ReAct 格式：
  - `Thought: ...`
  - `Action: ...`
- 不要把工具无法执行的能力只写进 prompt。需要真实读写文件或调用接口时，应配套程序能力或工具能力。

## 工具规则

- `tools/` 下的函数应保持小而专注。
- 工具函数必须有类型注解和中文 docstring。
- 工具返回值应是字符串或易于 prompt 消化的简单结构。
- 工具内部如果调用外部 API，应清楚处理失败情况，不要让异常信息泄露敏感配置。

## 文档规则

- 复杂功能先写设计文档，再写实施计划。
- 设计文档放在 `docs/superpowers/specs/`。
- 实施计划放在 `docs/superpowers/plans/`。
- 文档应记录明确范围，尤其要写清楚哪些改动暂不执行。

## Git 与工作区规则

- 当前仓库可能存在未提交的用户改动。开始修改前先查看 `git status --short`。
- 不要使用 `git reset --hard`、`git checkout --` 等破坏性命令，除非用户明确要求。
- 如果需要提交，只提交当前任务相关文件；如果工作区混有大量无关改动，应跳过提交并说明原因。

## 完成标准

完成任务前应确认：

- 相关测试已通过。
- 没有修改超出任务范围的文件。
- 新增或修改的非测试 Python 公共类/函数/方法有合适 docstring。
- 涉及记忆功能时，索引与实际文件状态一致。
- 向用户汇报时说明改了什么、验证了什么，以及任何残余风险。
